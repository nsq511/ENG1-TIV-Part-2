Architecture Characteristics 

Note: The characteristics in bold are from the ISO 25010 standard on software quality.

Requirement IDs are taken from the requirements referencing system on the website.

Characteristic 

Description / relation

Non-Functional Requirement IDs

How it will be achieved

Availability

Server downtimes during updates and ease of access to the application.

Distributed as a single download executable application with no online/external dependencies.. Assessment 2 updates can be made by refactoring and extension.

Deployability

Ease of distribution and running across hardware or systems.

Single system application with minimal hardware requirements. No server dependency.  

Extensibility

Ease of adding new features. Not very applicable to the project as the client only requests a standalone game

CR_ENGINE

CR_HARDCODED

ECS pattern allows new entities, components or systems to be added by modifying existing game logic (adding new components to existing systems or building new ones)

Fault tolerance 

Ability to shut down or recover from errors safely

NFR_RESILIENCE

Reset game or restart logic safely via persistence layer after crashes or or other failures.

Maintainability

Ease of fixing/updating without breaking other systems

NFR_RESILIENCE

Monolithic closed layered structure only allows each layer to access the layer below improving modularity and maintainability.

Modularity 

Distinct classes/packages with low coupling.

The layered structure means there are dependencies between layers but some bypasses are permitted if necessary. ECS system allows for minimal package modularity and grouping.

Performance

Execution per frame and rending times.

NFR_PERFORMANCE

2D game with hardcoded map and 16x16 tiles minimises processing requirements. AABB collisions only update when needed improving performance.

Reliability 

Systems ability to function as expected without runtime errors or bugs/exploits

NFR_RELIABILITY

CR_CHEAT

The game loop and timer are player tested for consistency (stress testing timer during events and pause). Rest logic allows for safe recovery.

Security

Levels of access to user and protection of system during system execution. Data encapsulation.

CR_FORBIDDEN

OOP-based game engine and language used, with data encapsulation allowing a secure API.

No online dependencies reducing external risks.

Scalability

Ability to adapt to larger engines/maps or extensions to larger systems with minimal redesign

OOP engine, ECS and layered structure supports system and component re-use for larger implementations. Game engine and 2D nature makes hardware and software scalability demands minimal.  

Testability

Ease of troubleshooting and player testing and level of user/product disruption during testing.

CR_ENGINE

Each ESC system can be tested independently. Each layer can be tested for cohesion and updated without affecting other layers.

Timing

Responsiveness to user input and system accuracy.

NFR_PERFROMANCE

Fixed / limited game loop and map size allows for predictable collision handling and framerates. 16x16 tiles opposed to 32x32 for sprites improves rendering.

Useability

Clarity of UI and player responsiveness

NFR_USABILITY

NFR_UI

NFR_INSTRUCTIONS

CR_RESOLUTION

UI, score, timer and sprites shall be simple, intuitive and unambiguous.
